Desired workflow (the WHY): today `bl close` delivers the squash to LOCAL `main` and the tracker auto-pushes the STORE (`balls/tasks`) to origin, but the CODE `main` is NOT pushed — delivered work sits on local main until a manual `git push origin main`. Hit live 2026-06-27: bl-b9a6 landed + was deployed locally yet stranded unreleased on local main. The release path then needs several manual touches. Goal: make delivery PROPAGATION and release PREP automatic, while keeping a single, explicit, deliberate human gate on what actually ships.

TARGET END STATE
1. CLOSE AUTO-PUSHES. After the delivery squash lands on local `main`, `bl close` pushes `main` to the code remote (`origin`) automatically — symmetric with how the store branch already auto-syncs. No-op when there is no remote (stealth). Delivered code reaches origin with zero manual steps.
2. RELEASE AUTO-PREPARES. The push triggers release-plz (already wired) to open/refresh the release PR (version bump + CHANGELOG) automatically — no human action to PREPARE a release.
3. SINGLE-TOUCH RELEASE GATE. The actual crates.io publish requires ONE explicit, deliberate human action (merge the release PR / one click), and that single touch RELIABLY publishes — no fallback rituals. Releasing stays a conscious decision; preparing it is frictionless.

WHAT THIS IS NOT: not full auto-publish on every close (the gate is deliberate — the maintainer decides what ships and when), and not the current multi-step ceremony.

DESIGN SEAMS TO ATTACK before building (problem statement, not a solution):
- WHERE the code-main push lives: extend bl-delivery's close.post (mirrors the tracker's store push) vs a new close.post action. The code remote is the repo's `origin` — git owns it (bl writes bl's files, not `git remote`); resolve it consistently, no-op in stealth.
- PUSH-RACE / FAIL-SOFT: the delivery already landed LOCALLY before the push, so a rejected push (remote moved, or a history rewrite like Wise's) must NEVER abort/rollback the close or lose the delivery. Fetch + ff/rebase + retry, or leave local ahead with a loud "push pending" signal. This is the SAME reconcile surface as the multi-clone-lag ball bl-c3c0 — solve them coherently.
- SEVERABILITY: auto-push is a DEFAULT that disables cleanly in stealth and via a config knob (for batch-push workflows). Removing the default should delete config, not edit core.
- HARDEN THE GATE: the current release-plz merge is finicky — `gh pr merge` sometimes does not fire the publish workflow (empty-`balls:`-commit fallback), the `[minor]`/`[major]` marker self-matches prose, `gh pr edit` is broken (REST PATCH needed). The single touch must publish every time WITHOUT these workarounds. Consider whether PR-merge is the right gate or a lighter trigger (a label / `workflow_dispatch` button) is cleaner.

DELIVERABLE: `bl close` code-main auto-push (core, severable, fail-soft) + a hardened release-plz config so one deliberate human action reliably ships a release. Relates to bl-c3c0 (shared push-reconcile surface).
