Design ball — deliverable is a tracked doc under docs/design/ named for this ball, converged by dialogue. Gated --needs on the query-surface design ball: the maintainer's read (verbatim, 2026-07-01) is "this is actually tied pretty deeply to [the query surface]. With a better query surface, this kind of thing falls out pretty readily."

PROBLEM. A crashed or abandoned agent holds its claim forever. The schema has `claimant` (a bare string) and `updated` (restamped by EVERY update, so it is not claim time); there is no stored claim timestamp, no lease, no TTL, no derived "stale" state. Recovery today is the manual orphan-takeover: commit WIP, tag, `bl unclaim` (no identity check), re-claim, cherry-pick. For unattended fleets this is the #1 operational hole: readiness is a query, staleness is not.

NUGGETS ALREADY IN PLACE. Claim time already EXISTS — it is the claim commit's timestamp in store history (derive, don't store; do not add a frontmatter field). Occupancy is deliberately cooperative: `unclaim` checks no identity, which is what makes takeover possible at all — but also means any buggy agent can silently steal a held ball.

CONSTRAINTS. Staleness is a DERIVED view (like status); any threshold ("stale after 2h") is policy and belongs in config/plugin, never core. A reaper, if one exists, is a plugin (prime.post is the natural hook). Mind machine-locality: the dead agent's `work/<id>` worktree lives on ITS box — a reap from another box cannot tear it down; committed WIP on the dead box's branch is stranded there until that box prunes. State whether that is acceptable.

QUESTIONS THE DOC MUST SETTLE. (1) Where claim-age derivation lives and what it costs (one git-log walk per claimed ball; the claimed set is small). (2) Surface: an age column on `bl list -s claimed`, per the query-surface design. (3) Reaping: plugin on prime.post vs a dispatcher-side loop vs nothing-in-core (document the takeover as the paved path). (4) Attack cooperative occupancy: is an identity-checked unclaim ever right, or does it just break takeover while providing advisory-only safety? (5) Does a reap unclaim or close? (unclaim — an empty close would archive work someone may still want).
