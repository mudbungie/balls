SS16 runbook step 2 assumes "bl prime founds substrate + an empty store; bl import --legacy fills it". Actual against the realistic case - a shared origin still carrying the legacy JSON store on balls/tasks:

- prime clone-in ADOPTS the legacy branch as the greenfield store (store checkout = the legacy commit: .balls/ present, no tasks/) - the SS2 adopt rule firing indiscriminately on a non-greenfield history;
- bl-delivery then aborts the prime (`No such file or directory (os error 2)` in the log; surfaced as the generic plugin-aborted error), and re-running prime hits the same abort - the SS12 no-op-converge property is lost;
- the default --legacy spec (balls/tasks:.balls/tasks) is also unusable in a fresh clone: only origin/balls/tasks exists, so it dies `fatal: Not a valid object name balls/tasks` (explicit --legacy=origin/balls/tasks works).

The cross-tracking trial hit the same wall ("legacy store aborts prime"). Migration of any shared repo needs either a legacy-shape guard at adopt (refuse or quarantine a tasks_branch with no tasks/) or a runbook + shim that survives the collision.
