larabail's windows-port (PR #365, commits c36b5030 + f7e0b779) made balls build AND test-compile on Windows by cfg-gating every Unix-only item:
  - delivery_repo.rs: is_executable() cfg-split (unix reads mode bits, windows returns true)
  - registry.rs: symlink cfg-split (unix symlink / windows symlink_file)
  - test sidecars: #![cfg(unix)] on files needing executable shell scripts/perm bits, + #[cfg(windows)] panic-stubs so helper symbols (bind_script, unpushable_remote) still resolve

Since it merged, main moved a lot underneath it — delivery_repo.rs was DECOMPOSED into delivery_repo_acts.rs (bl-43df), reads/plugin/lifecycle decomposed, many inline tests moved to new _tests.rs sidecars, and the bl-chore plugin + new integration tests landed. Any of those could reintroduce an ungated 'use std::os::unix' / .mode() / bare symlink / executable-bit test fixture that re-breaks the Windows build or test-compile.

DELIVERABLE: audit current main for Unix-only items not behind #[cfg(unix)] (or paired with a #[cfg(windows)] arm), and amend whatever drifted. The invariant to restore: 'cargo check' compiles on Windows and 'cargo test' compiles (not necessarily passes) on Windows — every os::unix path is conditioned, every test depending on exec-bit/shell-script is file-gated with a windows stub for any symbol it exports.
