Design ball — deliverable is a tracked doc under docs/design/ named for this ball, converged by dialogue. The biggest model-level gap: agentic work constantly spans repos (API change + client update; plugin + host), and balls cannot express it.

PROBLEM. Today a ball's delivery is one worktree in one repo, and §12 federation is many clones of ONE project naming one shared store — never many projects. Cross-store edges are impossible (a ball in repo A cannot --needs a ball in repo B). The one working recipe — satellite repos origin-repointed at a center store — is trap-laden: the global XDG task-remote shadows every other repo's own store (broke ~/dev/brazen, 2026-06-27), enrollment is multi-step, and a legacy store aborts prime. Maintainer (verbatim, 2026-07-01): "getting everything configured right is so painful that even I find it tedious."

NUGGETS ALREADY SHIPPED — the building blocks. (1) Task.root_commit (bl-1ce7): create stamps the checkout's root commit (git rev-list --max-parents=0 HEAD), claim REJECTS when the ball's root differs from this checkout's — remote-free repo identity, already reserved in the schema and riding the bedrock round-trip. (2) The ONE §12 remote ladder with a per-checkout task-remote binding and the stealth sentinel. (3) install as the consent-gated config-adoption verb.

THE REFRAME TO ATTACK FIRST (subtraction candidate). A single shared store serving N projects already MOSTLY works mechanically: balls are files; root_commit stamps which project each ball belongs to; the claim guard already refuses wrong-checkout claims; and cross-project --needs edges come FREE inside one store. The remaining gaps may be only: (a) READINESS IS NOT ROOT-AWARE — bl list in repo A shows repo B's balls as ready, but claim refuses them; once stores are shared, ready is checkout-relative and list doesn't know (filter ready to this checkout's root + rootless balls?); (b) ENROLLMENT ERGONOMICS — one command to point a satellite at a center (task-remote + config adoption), instead of today's multi-step recipe.

MAINTAINER'S SKETCH + DOUBTS (verbatim): "Make a config branch able to declare what root commits it operates against, and then flip the local store installation to trust that conclusion, letting a big-bang team task tooling implementation to run all at once? Feels a little bit like a risk of malicious config (kinda, it's a visible install process from a trusted source), and it runs afoul, or at least forks harder, off the non-git local implementation."

QUESTIONS THE DOC MUST SETTLE. (1) Does config declare a root→project mapping at all (config already NAMES the store; would it also name the fleet of roots?) — or is the mapping SUBTRACTED because root_commit alone routes and the stamps on the balls ARE the fleet (no second representation to drift)? (2) Root-aware listing semantics: what does bl list show in a shared-store checkout — this root's balls only, with a flag for the whole store? (3) Enrollment: what does `bl prime --center URL`-shaped one-shot adoption look like, and is install's existing consent gate sufficient against the malicious-config worry (§0: all config is potential RCE; the copy is explicit and visible)? (4) The stealth/local-only story: a shared center is inherently remote — what does cross-repo mean for a store with no remote, and does the design fork there or degrade cleanly? (5) Worktree placement for a ball claimed from a satellite: delivery territory is keyed by invocation path — confirm nothing else assumes store-repo == code-repo.
