Doc-vs-as-built drift found by verification; each wants a SS15-disciplined correction (or a small impl alignment where the doc is the better design):

- SS1: binding.toml does not exist on disk - the binding is a per-op wire payload only (layout.rs binding() has zero production callers).
- SS1: plugins/tracker/<pct-enc-remote>/ is never created; the tracker fetch/pushes by URL from each clone bundle own store repo, so "two clones sharing one store remote share one tracker dir" has no on-disk counterpart.
- SS1: unified log records omit the `phase` field (not null) on begin/done/seal/abort lines; only invoke/plugin-stderr lines carry it.
- SS4: "log_level is ... NOT a default-config/ seed entry" - the shipped seed balls.toml literally contains log_level = "info" (and tasks_branch).
- SS4: the XDG list-compose layer for [hooks] actually lives in an undocumented $XDG_CONFIG_HOME/balls/plugins.toml; a [hooks] table with _prepend/_append/_ban directives in the documented config.toml is SILENTLY ignored (trap: should compose or error).
- SS6: the seeded default schedule includes import.post = ["tracker"], absent from the SS6 table (bl-e614 staleness).
- SS7: binding.remote is omitted from the wire on origin-tier repos (present only with --remote or XDG task-remote); spec lists it unconditionally - a plugin cannot read the resolved remote in the common case. Also "current_state (null on create)" is realized as key-absent, and the two shipped plugins disagree on protocol self-description shape (tracker: 1, bl-delivery: [1]) - both legal per "<version(s)>" but worth one convention.
