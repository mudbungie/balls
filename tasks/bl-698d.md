WHY: §13's own text — the fixpoint "converges in ONE pre pass in the common case (the dial
holds), since plain prime never rewrites tasks_branch — that crosses only by install (§12)".
The tracker's prime/pre is spec-bound warn-only ("it NEVER rewrites tasks_branch"). So no
CONFORMANT plugin can ever move the dial: passes 2..FIXPOINT_CAP exist only to accommodate a
plugin violating the consent model. prime --install never uses the loop (it is sequential:
prime → install → prime). Mechanism without a conformant consumer — the same crime that
killed field_changes (bl-3bfd) and `bl plugin` (bl-587f).

SUBTRACT: run prime/pre once, materialize(tasks_branch), and if the configured name MOVED,
ABORT — "prime/pre may not move tasks_branch" — fail-not-silent, the §6/§8 cap disposition.
This ENFORCES the consent rule instead of tolerating 31 violations of it. Engine::fixpoint
and FIXPOINT_CAP go; bl-33db's runaway becomes a first-pass error.

Accepted delta: prime --install is now more deviant from plain prime's shape — a smaller
compromise than carrying a loop no conformant plugin can drive.

Deliverable: spec §8/§13 + §15 entry (bl-0a23/bl-33db entries gain supersession markers);
src/lifecycle_diffless.rs, src/checkout.rs + tests.
