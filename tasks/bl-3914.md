PROBLEM: every mutating verb writes the JSON op-log ({"ts":…,"lvl":"info",…}) to stderr alongside the terse human confirmation (e.g. 'claim bl-xxxx'), and every read verb writes the op-log to stderr too. The two share one stream, so an agent or script that wants clean output must wrap EVERY call: reads need '2>/dev/null', mutators need '2>&1 | grep -v "^{"' to keep the confirmation but drop the log. This was pervasive boilerplate across the bl-1b49 demo driver.

The op-log is a deliberate machine contract, so the fix is not to remove it but to make it suppressible: a global '-q/--quiet' (or 'BALLS_QUIET=1') that silences the JSON op-log while preserving the terse stderr confirmations and stdout. Then 'bl -q claim X' yields just 'claim X' with no grep dance. Severability: the default (log on) is unchanged; quiet is opt-in.

Observed during bl-1b49 E2E demo.
