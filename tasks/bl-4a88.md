## Undesirable behavior
`bl prime --stealth` happily founds a local-only tracker in a directory that is
**not** a git repo, and `bl create` / `update` / `list` / `show` / `sync` then
all work — but `bl claim` and `bl close` fail hard with a raw git fatal. Since
`close` is the only way to retire a task, you can mint tasks you can never claim
or close, and nothing warns you until you try to claim.

## Repro
1. `cd` into a directory that is not a git repo.
2. `bl prime --stealth --as X`  -> succeeds; founds the substrate.
3. `bl create "t"`              -> succeeds; prints the id.
4. `bl claim <id>`             -> FAILS: `plugin bl-delivery aborted ... git worktree prune: fatal: not a git repository`
5. `bl close <id>`             -> FAILS: `bl-delivery: git symbolic-ref --short HEAD: fatal: not a git repository`

## Where the surprise comes from
Core task storage (the `tracker` plugin) lives in its own git substrate under
XDG and is fully repo-independent — which is why create/update/list/show/sync
work. But the default hook schedule wires `bl-delivery` into
`claim`/`close`/`unclaim`/`prime.post`, and that plugin assumes the invocation
path is a code repo: it runs `git symbolic-ref` / `git worktree` against it. In
a non-git dir the first such call aborts and the raw git fatal surfaces. The
failure is *deferred* because `prime.post` bl-delivery only re-materializes
*still-claimed* tasks (`delivery.rs`) — zero on a fresh found, so no git call
runs until the first `claim.post` (`git worktree prune`, `delivery_repo_acts.rs:14`).

## Why it's undesirable
- `prime --stealth` (local-only, no remote) reads as a blessed standalone mode,
  so founding a tracker in a non-git dir feels supported.
- The failure is deferred and opaque: nothing at prime or create time signals the
  lifecycle is unusable; you only find out at `claim`, after filing tasks.
- A raw `fatal: not a git repository` from a plugin is a poor error for what is
  really a "this tracker isn't backed by a code repo" condition.
- `close` is the ONLY retirement, so in this state the tracker is structurally
  un-drainable.

## Decided approach (2026-06-23) — staunch the bleeding
The bug is narrow: **bl-delivery has a precondition — a git worktree at the
invocation path (`Project.root`, the §7 invocation path) — that it discovers
lazily, at the worst moment, in git's voice.** The non-git dir is just the
empty-input instance. Fix = make that precondition explicit and early, *in the
capability*, off ONE predicate (`git rev-parse --is-inside-work-tree` on `root`):

- **prime.post**: if `root` is not a git worktree, emit a single warning
  (plugin stderr → relayed in balls' voice, `plugin.rs:163`) and **no-op — do
  NOT abort prime**. Fires at founding, before any task is filed. Matches the
  house style: prime already *warns* (does not refuse) when nothing durable
  backs an ephemeral remote.
- **claim.post / close.pre**: the same predicate **aborts with that clean
  balls-voice message** instead of the raw `fatal: not a git repository`.

The message names **both drains**, so the tracker is never un-drainable:
`git init` to enable delivery, OR detach delivery for pure tracking
(`bl conf remove …`, or the one-command ergonomic bl-78e1 once it lands).

Located in **bl-delivery, not core prime** — severability: removing the delivery
default deletes config, it must never teach prime a "git repo or not" branch.
One predicate, two surfacings; any precondition-bearing plugin can self-report
the same way. Test in a non-git tmp dir under `tests/` (asserts the prime
warning and the clean claim abort; tarpaulin ignores `tests/`, so add a unit
angle on the predicate too).

## Rejected alternatives
- **Implicitly drop bl-delivery at the first prime in a non-git dir.** Bakes a
  transient *circumstance* (cwd-not-a-repo-right-now) into durable committed
  config (`plugins.toml`); drifts silently the moment you `git init`; collapses
  circumstance into *declared intent* (cf. stealth's `task_remote = none`, which
  is deliberate); puts delivery policy into core; and breaks prime's idempotent
  convergence (re-add on a later in-repo prime → config flips under you; don't →
  permanent stale strip). "First prime" is a bootstrap special-case = a missing
  reframe. The genuine pure-tracker is served by the explicit ergonomic (bl-78e1),
  which *declares* the intent in one command.
- **A `--no-delivery` flag / auto-drop on `--stealth`.** New flag = smell;
  stealth (no remote) is orthogonal to delivery (you can want stealth *inside* a
  git repo); a silent no-op claim loses claim's whole product (the worktree
  path) for the common "forgot `git init`" case.

## Seen in the wild
2026-06-23, against a non-git docs/ops directory. `bl conf` there showed
`claim`/`close` gated by `bl-delivery` and `create`/`update` by `tracker` only,
matching the split above. Workaround: `bl conf remove <op>.<pre|post> bl-delivery`
to detach delivery from the lifecycle verbs.
