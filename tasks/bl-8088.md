`src/seed.rs` embeds the default-config (`include_str!` of `default-config/{balls,plugins}.toml`) and, on first prime, MATERIALIZES it into `$XDG_CONFIG_HOME/balls/default-config/`, then treats that folder as an override that WINS if present. Bug: the materialized copy is written ONCE and never refreshed, so when the embedded default moves on (drop verb deleted [bl-65e0]; tracker→bl-tracker [bl-27bf]) the stale XDG copy silently shadows the current embedded default.

Observed 2026-06-27: `~/.config/balls/default-config/plugins.toml` (Jun 7) still has `drop.post`, `claim.pre`/`unclaim.pre`, and `tracker`; core's embedded/shipped default has none of these (uses `bl-tracker`, no drop). Every NEW repo founded on this machine seeds from the STALE copy.

Same failure class as bl-d081: an auto-written XDG file silently winning over the correct LIVE source.

Fix direction (attack before building): don't auto-materialize the embedded default into XDG. Use the embedded default directly when no override exists; treat `$XDG/balls/default-config/` as a DELIBERATE, never-auto-written override (present only if the user authored it). "Don't store what you can compute" — materializing a copy of an embedded default just creates a duplicate that drifts. Severable: removing the auto-materialize deletes the drift with no behavior change for users who never customized.

Workaround for an affected machine: `rm -rf ~/.config/balls/default-config` → falls back to the current embedded default.

Core's SOURCE is correct; this is purely the seed/drift mechanism.
