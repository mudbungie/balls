Found by the bl-1b49 E2E demo (multi-agent parallel claims).

REPRO: two physically-simultaneous `bl claim` invocations on DISTINCT ready tasks against the SAME local clone (same project path → same XDG clone). They serialize on git's store-branch ref-lock; one wins (sealed), the other loses. The loser has NO retry, so the claim fails outright. In ~7/12 runs the loser's bl-delivery pre-hook has ALREADY written `claimant` into tasks/<id>.md in the clone working tree before the seal's `git merge --ff-only` aborts — leaving that file modified and STAGED in the clone.

TWO consequences:
1. Wedge — every subsequent bl op fails 'Your local changes to tasks/<id>.md would be overwritten by merge' (ff-only refuses a dirty tree).
2. Phantom claim — `bl list`/`bl show` read the uncommitted claimant from the dirty working tree and report the task as claimed, though it was never sealed, committed, or pushed and the loser has no work/<id> worktree. Derived status is corrupted.

The winner is never corrupted, so the 'no claim clobber' guarantee holds; this is a robustness/atomicity gap on the LOSING path, not a clobber.

RECOVERY: `git -C <clone>/tasks reset --hard` (NOT `checkout -- .` — the change is staged). Winner's sealed claim survives.

FIX DIRECTIONS: a lost ref-lock race should roll the clone working tree back atomically (the seal is already a guarded merge — extend the failure path to reset the worktree), and/or bounded retry on the ref-lock loss (cf. the ETXTBSY busy-retry precedent). Stats: 12 simultaneous races → loser never succeeded; 0 both-ok, 5 clean-fail, 7 wedged.
