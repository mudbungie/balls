## Undesirable behavior
`bl conf` (full dump) lists `drop.post` as a resolved hook key, but **every
per-key operation on it is rejected** — read (`bl conf drop.post`), `set`,
`append`, `prepend`, and `remove` all fail with `unknown key 'drop.post'`
(exit 1). The full-dump path and the per-key path disagree about whether
`drop.post` is a valid key. And `drop.post` is a **default-seeded** hook entry
with no user-facing `drop` verb, so it cannot be inspected or edited through the
documented `bl conf` interface at all.

## Repro (a default-seeded tracker — e.g. `bl prime --stealth` in a fresh dir)
```
$ bl conf | grep drop
drop.post     bl-delivery, tracker  landing

$ bl conf drop.post
bl: conf: unknown key 'drop.post' — keys: task-remote, task-branch, log-level, <op>.<pre|post>, show, list   (exit 1)

$ bl conf remove drop.post bl-delivery
bl: conf: unknown key 'drop.post' — ...   (exit 1)

$ bl conf set drop.post tracker
bl: conf: unknown key 'drop.post' — ...   (exit 1)

$ bl conf append drop.post tracker
bl: conf: unknown key 'drop.post' — ...   (exit 1)

$ bl help | grep -i drop
(nothing — no user-facing `drop` verb)
```

## The inconsistency
The dump path treats `drop.post` as a normal `<op>.<pre|post>` key and prints it;
the per-key validator's op-allowlist (`task-remote, task-branch, log-level,
<op>.<pre|post>, show, list`) doesn't recognize `drop`. One side is wrong: either
`drop` belongs in the op-allowlist (so the seeded entry is inspectable/editable),
or the dump shouldn't surface `drop.post` at all (and the default shouldn't seed
a hook in a form users can't manage). Either way, the read path and the
get/set/append/prepend/remove path should agree on the valid key set.

## Expected (leaving the fix open)
`bl conf`'s dump and its per-key get/set/append/prepend/remove should agree on
which keys exist. A hook entry present in the default config should be reachable
via `bl conf` — editable, or explicitly surfaced as read-only/internal. Not
prescribing which.

## Context
Found 2026-06-23 while detaching `bl-delivery` from a non-git tracker's lifecycle
(workaround for #370). `bl conf remove` cleared `bl-delivery` from
`claim`/`close`/`unclaim`/`prime`, but `drop.post` was the one default-seeded
entry it refused — leaving a residual `bl-delivery` reference clearable only by
hand-editing the landing `balls.toml`.
