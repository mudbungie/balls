Symptom: install::wiring::tests::a_failing_pre_plugin_aborts_the_seal_and_cleans_up flakes ~2-in-8 under parallel cargo test (passes 6/6 in isolation). Not a test-isolation/env race — a production ordering sensitivity.

Mechanism (src/plugin.rs:230): spawn() does `child.stdin.take()...write_all(payload)?` then `child.wait()`. A plugin that rejects an op by exiting BEFORE draining stdin (the fake tracker is `#!/bin/sh; exit 1`) closes the pipe under the parent write. Usual timing: payload lands in the 64K pipe buffer first, write_all Ok, wait() returns exit 1 -> "tracker plugin failed" abort. Under load: child exits first, write_all hits EPIPE, the `?` returns "Broken pipe (os error 32)" and wait() is NEVER reached -> the plugin exit status (and its name) is masked, nondeterministically.

This contradicts the §7 contract (plugin.rs:201-205): the exit status is the authoritative signal; payload delivery is not. EPIPE on delivery must not preempt it.

Fix: swallow io::ErrorKind::BrokenPipe from the payload write (a plugin closing stdin early is its right); any OTHER write error still propagates. Let child.wait() be the single source of truth. Extract the write into a small free fn (feed) so the swallow / propagate / ok branches are unit-coverable with mock writers, deterministically. The flaky integration test then passes with no change to it (write swallows EPIPE -> wait() returns exit 1 -> err contains "tracker").
