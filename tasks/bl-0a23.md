Design converged 2026-06-08 (dialogue under bl-fa00). Replaces the eager
found-orphan-then-reset approach (landed under bl-fa00) with LAZY
materialization, so the "unrelated histories" divergence is never CREATED
rather than reconciled after the fact. Supersedes bl-fa00's tracker reset.

## Core invariant — "a branch is a disk path"

Core's only store primitive is `materialize(config.tasks_branch)` — ensure that
branch has a worktree on disk, founding an orphan (`tasks/.gitkeep` root) IFF the
ref is absent. Core knows nothing of remote / adopt / found / promote / fork.
`prime` founds the LANDING (`balls/config`) eagerly as today; the TASKS branch is
materialized only by this primitive.

## Prime is a bounded fixpoint — core owns the loop, NOT the plugin

```
loop:
  materialize(config.tasks_branch)
  run prime/pre        # tracker settles the NAME -> may rewrite config.tasks_branch
  if config.tasks_branch unchanged: break
run prime/post         # tracker settles the CONTENT: fetch+ff (adopt) and push (publish)
```

The config edit IS the loop signal — core diffs `tasks_branch` across the chain;
NO §7 return channel needed (the dial lives in the config branch core owns).
Convergence is "the dial stopped," not "the bl-7110 guard tripped."

## Tracker sits in two slots (one per axis)

- **prime/pre = settle NAME**: read the remote's `balls/config` `tasks_branch`, set
  local `config.tasks_branch` to the canonical name. 0 passes in stealth or when
  it already matches; 1 extra pass on a remote rename.
- **prime/post = settle CONTENT**: fetch+ff (adopt remote commits) and push
  (publish local). Both halves no-op when they should — push is a no-op-ff after
  an adopt; found-on-miss fetch finds nothing and the push CREATES the branch.

## Every story = a sequence of (dial-move) + (materialize)

| Story | pre (name) | core loop | post (content) |
|---|---|---|---|
| stealth | no remote, no change | 0 passes | nothing |
| bootstrap / promote stealth | keep my name | 0 passes | push (creates remote) |
| join existing | already my name | 0 passes | fetch+ff (adopt) |
| adopt under remote's name | rewrite to N | 1 pass | fetch+ff (adopt) |
| fork (explicit non-default name) | keep my name | 0 passes | push (creates new branch) |

Empty skeleton AND populated-stealth-store are the SAME path (push whatever local
store I have).

## Invariant is per-clone, not global

`config.tasks_branch` is the single source of truth for which branch THIS clone
uses; the remote may host MANY tasks branches. "One tasks branch" is the common
case, not a law.

## Fork vs footgun discriminator (already in code, keep it)

`store_elsewhere` only warns when `tasks_branch == DEFAULT` ("user-set or
install-adopted — not the silent gap"). An explicit non-default name = deliberate
fork (proceed silently); the default name + a remote store elsewhere = likely
misconfig (warn — decide later if it hardens from warn to a confirm).

## Config branch always materialized + synced

Even when functionally idle: buys symmetric branch addressing and keeps `install`
a pure-filesystem op for core while the tracker's `install.pre`/`install.post`
sync hooks carry the remote half (consistent with the wired `fetch_config` on
`install/pre`).
