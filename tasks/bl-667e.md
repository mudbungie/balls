From the bl-1a66 conformance review (finding m1, minor). §7 post payload lists 'previous_state / current_state (null for create / close-drop respectively)' — for claim/update, current_state should be the SEALED after-state Task.

CURRENT STATE: OpContext.after is hardcoded None (mutate.rs:71), so the post wire's current_state is ALWAYS null. A post reactor cannot read the landed Task from the wire; it derives from metadata (bl-id trailer) + git.

This needs a decision, not a quick patch:
(a) POPULATE it correctly — the after-state must be the post-SEAL task (pre plugins may have edited it), so mutate.rs (pre-seal) cannot compute it; the Engine/Subprocess would have to re-read the sealed task and thread it onto the post wire — couples the generic engine to Task parsing.
(b) STRIKE it from §7 — consistent with §14 ('post never mutates the ball; derive-don't-store'): post reactors should derive landed state from git, making the wire field vestigial. But editing the FROZEN spec requires a §15 revision-log entry per the freeze discipline.

Low impact (no shipped plugin reads it). Recommend (b) — it matches the derive-don't-store invariant — pending the user/owner's call on touching the frozen doc.
