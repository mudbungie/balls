From the bl-1a66 conformance review (finding M3, major). §8: 'config / install keep the sealing shape but seal to the LANDING, not the store'; §14 governs it with reverse-order rollback + core un-seal.

CURRENT STATE (drift): standalone 'bl install' is unwired — lib.rs catch-all prints Op::new(other).plan() and returns Ok (the doc at lib.rs concedes 'install is unwired'). The only adoption path, prime --install → adopt::install_local, runs install.pre via a hand-rolled loop (no reverse-order rollback if a later step fails) and commits the landing with a bare 'git add -A && git commit' (commit_landing) — no change worktree, no atomic commit+integrate seal, no §14 un-seal.

DESIGN QUESTION to resolve before building (why this isn't a quick fix): install does NOT cleanly fit the task-op Author-first seal shape. The path-copy 'base change' needs the source ref PRESENT, but for prime --install the tracker FETCHES it in install.pre — so the fetch must precede the base copy, inverting Author→Pre. Options: (a) model standalone 'bl install' (source already synced) as a landing-sealing op via the engine, and keep prime --install as a composition (fetch, then the install op); (b) give the diffless engine a between-pre-and-post core-work slot (like prime's materialize) and route install_local there with proper unwind. Decide, then implement: wire the standalone verb (flag parsing --from/--to/--bin + dispatch) and route the pre-chain through the engine's reverse-order rollback. Keep 100% coverage.

Mitigation noting why this is major-not-critical: the adoption is local-only and git-recoverable, and the tracker's install.pre is an idempotent fetch whose rollback no-ops — practical blast radius is low; the gap is structural.
