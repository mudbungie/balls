Implementation ball (maintainer, 2026-07-01: "Probably can go straight to the implementation ball on this one, I don't think there's any ambiguity.")

PROBLEM. Progress notes ride `-m` on update commits; the manual's own words: "the journal IS git history (`git log -- tasks/<id>.md` in the store checkout under $XDG_STATE_HOME/balls/clones/…/tasks)". `bl show` renders none of it, so an agent taking over a ball cannot see the prior agent's notes without knowing the XDG spelunking incantation — an interaction gap exactly at handoff, where the notes matter most.

IMPLEMENT. Fold a journal section into human `bl show <id>`: the store-branch history of tasks/<id>.md, oldest-first, one entry per commit — timestamp, actor, and the note payload (render the §5 commit-message protocol's message, not the raw envelope/trailers). Follow the worktree-line precedent exactly: the human view folds derived state in; `--json` stays the lossless bedrock mirror of stored frontmatter and NEVER carries derived fields — the journal is derived (it is history), so it does not enter bedrock. Must work for closed ids too (show already reconstructs those from history — the journal is the same walk). Cost: one git-log walk over the store clone per show.

Notes: no new verb, no body-append flag, no stored field — this is a read-side rendering of state that already exists. If the entry list is long, oldest-first with no pagination is fine for v1; a cap is policy and can wait for the query-surface design.
