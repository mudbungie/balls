IMPLEMENTATION ball (spawned from investigation bl-d9b0). Goal: a clean `bl prime` stealth founding emits NO raw JSON to the terminal at the default log_level, while genuine warnings/errors stay legible.

== ROOT CAUSE ==
There is no human renderer for the op log: `Log::record` (src/log.rs:124-133) echoes the *full JSON line* straight to stderr (`eprint!("{line}")`, src/log.rs:132) for every record at-or-above the threshold. The threshold defaults to `info` (src/config.rs:58-66 default_log_level -> "info"; resolved in src/reads.rs:121-122, src/checkout.rs:120/168/184, etc).

The severity ladder has only THREE rungs and NO `warn` (src/log.rs:52-57 enum Level{Debug,Info,Error}). By the design doctrine (src/log.rs:45-51), core narration is `Debug` (quiet at default) but "a plugin speaking (enveloped stderr) is Info". So EVERYTHING a plugin writes to its stderr collapses to one level, `Level::Info`, set at the single relay site src/plugin.rs:195-199 (`relay` -> `capped_lines` -> `self.log.record(Level::Info, name, ...)`). At the default `info` threshold that is at-threshold, so it is echoed as JSON.

PRIMARY symptom: the routine W1 line fires on EVERY stealth founding — src/tracker/prime.rs:42-48, `eprintln!("tracker: store is stealth (local), not auto-syncing")`. The tracker writes it to its own stderr; core relays it line-by-line at Info; user sees one raw JSON envelope.

SECONDARY symptom (per-line JSON wrapping of git errors): the tracker funnels every git failure through one runner that returns an io::Error carrying git's MULTI-LINE stderr verbatim — src/tracker/git.rs:22-37 (`Err(io::Error::other(format!("git {}: {}", args.join(" "), <stderr>)))`). On a handler error the tracker binary prints it once: src/tracker.rs:68-75 `eprintln!("tracker: {e}")`. Core's relay then splits THAT multi-line blob on newlines — src/plugin_io.rs:48-57 `capped_lines` calls `sink` once per line — and each line becomes its own Info JSON record (src/plugin.rs:196-198). One git fatal -> N JSON envelopes.

The SKILL claim (SKILL.md:169-174) "at info routine ops are quiet ... what reaches stderr is a plugin speaking (warnings) or a failure" is misleading: a plugin has no debug rung, so its ROUTINE narration is indistinguishable from its warnings — both are Info — and both leak at the default level, as JSON.

== CHOSEN MINIMAL CHANGE (two small, independent levers) ==
LEVER A — demote the routine line so it stops leaking at default level. The cleanest fix is at the SOURCE, not the renderer: the "store is stealth, not auto-syncing" line (src/tracker/prime.rs:46) is routine-and-unconditional (fires on every stealth founding), unlike the three CONDITIONAL tracker warnings (prime.rs:51 ephemeral-remote W2, prime.rs:54 store-elsewhere, remote_ops.rs:74 legacy-quarantine) which are exceptional and SHOULD surface. Because a plugin has no per-line level channel (its stderr is uniformly enveloped at Info, plugin.rs:197), demoting one plugin line below info is not expressible today. Pick ONE:
  (A1) Simply DROP the W1 eprintln (prime.rs:43-48 keeps `return Ok(())`, loses the line). The declared-stealth fact is already re-derivable via `bl conf` (the landing task_remote sentinel, bl-9df0), so the line is purely advisory; the W2 ephemeral-remote warning still covers the "forgot to federate via --remote" case. Most subtractive. Attack: loses the "deliberately local vs forgot to federate" nudge on a plain stealth founding — but `bl conf` shows it on demand, and stealth founding is the EXPECTED first-run, so narrating it every time is the wart itself.
  (A2) Give the plugin a way to mark a line debug-level (e.g. tracker writes routine narration to stdout — which core inherits verbatim and does not envelope, plugin.rs:154-156/181 stdout=inherit — OR a "BALLS_DEBUG: " stderr prefix core maps to Level::Debug). More mechanism; only worth it if other plugins will need routine narration. Default to A1 unless the implementer finds a second consumer.
LEVER B — render plugin op-log lines as terse human text at info instead of raw JSON, and stop the per-line JSON explosion of multi-line errors. Today the terminal echo IS the JSON (log.rs:132). Change ONLY the terminal echo (keep the JSON-lines FILE sink exactly as-is — it is the §6 machine record and metrics source): when emitting to stderr, render a plugin (`src != "core"`) line as its `msg` text (optionally a one-char severity sigil), not the envelope. The file still gets the full JSON line; the terminal gets the human line. This also dissolves the secondary symptom for the human (still N file records, but the terminal shows readable text), and at debug the user can still ask for raw if desired — though simplest is: file=JSON always, terminal=human always. Keep the Error records (plugin.rs:149/211, the failure locus) loud and legible.

RECOMMENDATION: do A1 (delete prime.rs:46 line) for the primary symptom — it is the minimal, source-side fix and makes the exact repro go quiet. Do B as the principled fix for "JSON should never be the human-facing terminal output" and the secondary multi-line-git-error symptom. They are independent; A1 alone closes the reported repro, B alone makes ALL plugin output legible (and keeps the stealth nudge, just readable). Implementer may ship A1 first and B as a follow-up, or both.

== ALTERNATIVES CONSIDERED & REJECTED ==
- Raise the default threshold to `error`: silences genuine W2/store-elsewhere/legacy warnings the user needs. Rejected (violates must-not-break).
- Add a `warn` rung to the Level ladder: large change to a deliberately 3-rung ladder (log.rs:45-57, SKILL.md:172-174 "there is no warn"); does not by itself stop JSON rendering. Rejected as over-mechanism.
- Suppress ALL plugin Info echo at the terminal (file-only): silences the legitimate conditional warnings. Rejected.
- Have the tracker collapse multi-line git stderr to one line at git.rs (join with "; "): treats the symptom in the wrong layer (the verbatim git stderr is a deliberate contention signal, git.rs:5-8) and still emits JSON. Lever B at the renderer is the right layer.

== MUST NOT BREAK ==
- Genuine warnings MUST remain legible to the user at the default log_level: the ephemeral-remote W2 (prime.rs:51), store-elsewhere (prime.rs:54), legacy-quarantine (remote_ops.rs:74) lines, and especially plugin-abort/rollback Error records (plugin.rs:149, plugin.rs:211) and core aborts. Do NOT blanket-silence Info plugin output.
- The JSON-lines FILE sink (clones/<enc>/log) is the §6 machine record + metrics source — keep it byte-for-byte JSON. Only the TERMINAL echo may change.
- 100% coverage + 300-line cap. Update SKILL.md:169-174 to stop claiming routine plugin lines are quiet at info (it is the renderer/level that makes them quiet now). prime_tests.rs / log_tests.rs / plugin_tests.rs will need updates.
