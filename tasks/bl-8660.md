Runbook step 5 force-pushes the greenfield orphan store over the published legacy balls/tasks ref — a non-ff rewrite that strands every clone's remote-tracking ref and exiles the legacy history to a balls-archive side branch. Replace it with a history join: in the XDG store checkout, git merge -s ours --allow-unrelated-histories the legacy tip (keeps the greenfield tree byte-for-byte, makes the new tip a DESCENDANT of the legacy tip), then a plain push. The push is a fast-forward; every clone fetches cleanly; the legacy history stays where it always lived — early history of balls/tasks — so balls-archive deletes from the runbook.

Why: strengthens the spec by subtraction — §16's 'the ONE deliberate force-push' exception deletes; bl never requires a ref rewrite, full stop. The operator's explicit consent becomes the local merge; the ordinary publish machinery (a plain fast-forward push) delivers the cutover — no special push command. An upgrade that rewrites no published ref is not a breaking change for downstream clones.

Touch points: runbook step 5, §16 'Branch & history', §12 quarantine paragraph ('cutover rewrites' -> 'cutover joins'; 'every store commit carries tasks/' becomes a tip property), e2e demo prose (bl-d754), release checklist, four code comments citing 'the runbook's explicit force-push'. Docs and comments only; quarantine/push logic untouched (it reads tip shape, and a joined tip carries tasks/).
