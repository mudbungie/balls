Design ball — deliverable is a tracked doc under docs/design/ named for this ball, converged by dialogue. Maintainer (verbatim, 2026-07-01): "That's an interesting problem. How do we solve it? I'm not even really sure what the right mechanism is. A socket that balls can signal to when relevant state changes occur? What's the right mechanism here? How does agent signalling happen?"

PROBLEM. There is no way for an agent (or a dispatcher) to LEARN that store state changed — a ball became ready, a claim released, a gate closed. The only loop is poll: prime/sync + list → claim → spawn, and every adopter rebuilds it from scratch. Same-box parallel claims additionally need harness-side serialization (bl-07d6: physically simultaneous claims — the loser now resets clean but still fails and must rerun). What already works: cross-clone claim contention IS atomic — the tracker's push is the compare-and-swap; a non-ff push aborts the op and rolls back.

THE LOAD-BEARING CONSTRAINT. "Git provides sync; there is no server" (§0). A socket implies a resident process — a daemon forks the core bet. The doc must either defend bending that principle or place signaling OUTSIDE core entirely.

DESIGN SPACE TO ATTACK, maximal-subtraction first. (0) NO new mechanism: ship a reference dispatcher (a contrib loop: sync; list --json; pick; claim; spawn — with the serial pre-claim discipline baked in). Polling is honest for a git-medium tracker; the gap is that N adopters write N loops, so ONE blessed loop may be the whole fix. (1) Local signaling is already free: every mutation runs op.post hooks, so a notify PLUGIN (write a line to a FIFO / unix socket / webhook) is pure policy, zero core change — but it only sees ops made from THIS clone. (2) Remote-originated changes only arrive at sync, so remote eventing belongs at the HUB: a post-receive hook / forge webhook on the store branch wakes the dispatcher — signaling lives off-box, no daemon on any agent box. (3) A `bl watch` verb (blocking, internally polling): packages the loop but adds a verb (smell) and a blocking op that forks the op model.

QUESTIONS THE DOC MUST SETTLE. (1) Which layer owns wake-ups: contrib dispatcher, notify plugin, hub hook — or a composition (plugin for local, hub for remote, dispatcher consuming both)? (2) What is the event payload — or is the event just "wake up and run bl list" (events carry no truth; the store is the SSOT, so a bare poke may be the elegant answer)? (3) Does anything here justify touching core at all? (4) Same-box claim serialization: dispatcher discipline documented as the paved path, or is there a cheap in-core fix (an flock around the store checkout) that dissolves bl-07d6-class races for everyone?
