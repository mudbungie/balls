Surfaced during bl-8f5b (shellback hardening audit, 2026-06-09). marker::strip in the issues plugin accepts a / inside a bl-… marker id parsed from a GitHub issue title, and that attacker-influenced id flows into store::read_ball / is_live path joins (tasks/<id>.md). A crafted title marker like [bl-../../x] yields a read outside the tasks/ dir — a READ-ONLY path-traversal shape, mirroring the core mirror-path guard fixed in 938e75a0 (bl-2d6d LOW-2).

Severity: low (read-only, the store checkout is semi-trusted local state, no write/exec) — but the fix is one validation: constrain the marker id grammar to the minted shape (bl- + fixed-alphabet suffix, no separators) at the parse seam, so nothing downstream needs to care. Add a hostile-marker test.
