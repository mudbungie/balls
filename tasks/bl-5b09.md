Design ball — deliverable is a tracked doc under docs/design/ named for this ball, converged by dialogue. Maintainer (verbatim, 2026-07-01): "Okay, how do we solve this?"

PROBLEM. install is a pure path-copy and "bin/ never travels" (§6) — deliberately: all config is potential RCE (§0), binding validates a live protocol, a dangling schedule entry prunes at seed / errors clean at dispatch, never code execution. The consequence: adopting a center's capability set is high-friction. The real ecosystem today — bl-adversary (close.pre review gate) and the gh forge/issues plugins — lives in sibling repos, hand-built, hand-placed beside bl, hand-wired. The MECHANISM is right; the ACQUISITION story doesn't exist. This indirectly caps every workflow objection ("no turnkey review flow"), because all workflow policy lives in plugins by design.

HARD LINE. Auto-fetch-build-bind is rejected by definition — that IS remote code execution, exactly what §0 forbids. Whatever the design, a human runs the acquisition command; balls never executes anything it fetched.

DESIGN SPACE, maximal-subtraction first. (0) Distribution is cargo's job: plugins are ordinary crates (`cargo install balls-adversary`); balls never acquires anything. The entire gap then reduces to DISCOVERY: the moments where balls already refuses — the dispatch-time "plugin not bound" error, and install's post-copy binding step — could carry an acquisition HINT (e.g. "bl-adversary is not installed beside bl; its schedule entry names source = 'cargo install balls-adversary'"). The hint is committed config TEXT, crosses via install like all config, is advice from a center you already consented to, and is executed by the human. (1) A doctor-shaped read surface: list schedule entries that are unbound, each with its hint (possibly just `bl conf` provenance output growing a bound/unbound column — prefer the existing verb over a new one). (2) First-party bundling: ship more capabilities in-crate as opt-in binaries (the bl-chore precedent) — pragmatic 80% for the common set, zero acquisition problem.

QUESTIONS THE DOC MUST SETTLE. (1) Where the hint lives — a field on the plugins.toml schedule entry vs the plugin's own config folder (core "knows two things about a plugin: name and binary path" — does reading a hint field violate that, or is a pass-through string core never interprets acceptable?). (2) Exact surfacing moments: dispatch error, install binding refusal, conf dump — which, and with what wording. (3) Trust framing to state explicitly in the doc: the hint's provenance is the installed config's provenance; installing config from a center is ALREADY the trust decision. (4) Is versioning/protocol-compat in scope (the binding step already validates protocol — is that enough)? (5) Whether first-party bundling changes the calculus enough that (0)+(2) is the entire design.
