Repro: legacy fixtures on a branch, `bl import --legacy` prints `import N balls` (nodes seal) then hangs forever (single thread in futex wait; `timeout 60` exits 124; 100% reproducible).

Root cause: src/lib.rs:207 binds `std::io::stdin().lock()` for the whole verb; the cutover button then runs the epic reciprocal-edge pass through `mutate::run` -> `Editor::live()` -> `io::stdin().lock()` again. The std stdin mutex is non-reentrant: self-deadlock.

Consequence: the cutover button is unusable, and a killed run leaves nodes imported with NO reciprocal edges, so every legacy epic migrates spuriously ready (the exact failure the SS16 edge pass exists to prevent). The mechanism itself is sound: manual `bl update <epic> --needs <child>` works, and the plain pipe `bl list --legacy --json | bl import` completes (it mints no edges, as designed).

Fix shape: do not hold the stdin lock across the edge pass (lock per read, or collect the records before dispatching the updates). Why it matters: SS16 sells import --legacy as PURE ORCHESTRATION of the pipe plus the edge pass; today it is strictly worse than the pipe.
