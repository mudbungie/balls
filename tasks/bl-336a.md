## Why

§12 states "each op runs against an up-to-date store (pull → mutate → push)" and E5 cites "the §13 pull→mutate→push contract" — but §13 defines no such contract and the pull half was never built: mutating ops wire the tracker only into *.post (push; src/tracker dispatch `(_, "post") => remote_ops::push`), and nothing in mutate.rs syncs first. The spec cites a contract that exists nowhere.

## Resolution (agreed 2026-06-09, by subtraction)

Re-derive the spec from the build: the contract is MUTATE → PUSH, optimistic. An op seals against the local store; a stale store surfaces ATOMICALLY at the push — the non-ff reject (E5) is the one-step detect-and-act contention check, and recovery is `bl sync` + retry. This is the same argument §13 already makes against a separate sync contention probe: a pre-pull would add a remote round-trip to every claim/close plus a TOCTOU window the ff-push reject closes anyway. Concurrency stays safe: a stale claim seals locally, the push rejects, the op rolls back — the remote store never sees the losing mutation.
