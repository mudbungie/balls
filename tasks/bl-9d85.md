The bl-d014 adversary close.pre review gate is 100%-covered only against a STUBBED claude (tests/cli.rs puts a fake `claude` on PATH). The CLI flags are verified real — `claude --print --model <m> --effort <e>` with the prompt on stdin all exist (effort only works with --print). What is NOT verified is the real end-to-end round-trip:

- Does a real model, given the gate's prompt, actually end its reply with EXACTLY `REVIEW: PASS` / `REVIEW: FAIL` and nothing after, so `src/review.rs::passes` (last non-empty line == `REVIEW: PASS`) fires correctly?
- Does `claude --print` (default text output-format) emit only the model text, with no trailing wrapper that would break the last-line parse?

Because the gate defaults to FAIL-CLOSED, the FIRST real use that doesn't parse a genuine pass will silently ABORT the user's close — a bad first impression for a shipped-but-dormant feature.

Deliverable: an end-to-end smoke test against real claude — build + install the binary beside `bl`, `bl conf prepend close.pre adversary`, close a trivial task that should pass (confirm exit 0 / close proceeds) and one that should fail (confirm abort). Since a real-claude call can't run in CI and tarpaulin ignores tests/, this likely lands as a documented manual test ritual (a script + README section), not a coverage-counted test. Capture the actual model output shape so the verdict parse is validated against reality, and tune the prompt if the model adds trailing text.
