WHY: bl close re-gates the FOLDED tree (bl-ee85) — the tree that actually lands — so a repo
whose pre-commit hook is expensive (this repo: clippy + full tests + tarpaulin, ~1 min) pays
it on every intermediate worktree commit AND once at close. The close-time gate is the load-
bearing one; per-commit gating is repo policy, not a bl invariant.

ADD one SKILL.md paragraph: intermediate commits in the work/<id> worktree may reasonably use
git commit --no-verify — the close gate re-runs the repo's hook on the folded tree before
anything lands, so the gate-of-record is never skipped. Per-repo hook cost stays the repo's
own policy decision; bl takes no position beyond stating where the gate-of-record sits.
