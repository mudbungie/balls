Confirmed drift surfaced while smoke-testing bl-0a23 (federation works only if you pass --remote or set XDG remote; plain "git clone <proj>; bl prime" goes stealth).

## The drift (two parts, both contradict §0/§12)

1. **Implicit origin discovery reads the LANDING, which is local-only and never has an origin.** `src/checkout.rs` `resolve_remote(...).or_else(|| origin_of(landing))`, and `origin_of` runs `git remote get-url origin` on the landing checkout. But the landing (`balls/config`) is §2 "path-derived, single-owner, install-transport" — `substrate::found_landing` creates it with a bare `git init`, and NO production path (`found_landing`, `adopt`/`install_local`, `seed`, the tracker fetches by URL) ever `remote add`s an origin to it. So `origin_of(landing)` is permanently `None` in real use — the `origin` precedence tier is DEAD, exercised only by two unit tests that manually `git remote add origin` on the landing (`checkout_tests.rs:76,137`).

2. **Core, not the tracker, does the discovery.** `checkout::resolve_remote` is called by core `bind()` (prime/sync) AND `mutate.rs:61` (create/claim/...). Reading a git remote is conceptually remote-resolution; §0 makes "remote-talk plugin-exclusive" and §12 houses resolution under "Tracker's prime". Implicit origin discovery in core is the structural drift.

## What it breaks (spec, literal)

§12: "**The STANDARD case needs no install:** the seeded defaults name `origin` + `balls/tasks`, so a fresh clone of a repo whose store sits there runs `bl prime; bl list` out of the box." The "fresh clone of a repo" is the PROJECT repo (the invocation path) — its `origin` is the real remote the code rides. Code never consults it (reads the landing instead), so the out-of-the-box path cannot fire. The default `balls.toml` has no `remote` field, so the precedence genuinely depends on origin discovery.

Lineage: predates bl-0a23 (came in with `resolve_remote`, bl-cd21). bl-0a23 only surfaced it.

## Fix direction

Explicit values (`--remote`/`--center`/XDG `remote`) stay core config-reads. Implicit `origin` discovery MOVES to the tracker and reads the **invocation_path** (the project repo), not the landing. The §7 binding already carries `invocation_path`, so the tracker can `git -C <project> remote get-url origin` when it has no explicit remote. Core then hands the tracker `remote: None` (no pre-resolved origin in the binding) and the tracker decides stealth-vs-tracked after its own probe.

## OPEN design topic (attack before building)

Moving discovery into the tracker ripples into the stealth gate: today every tracker handler short-circuits on `binding.remote == None` ("no remote => stealth"). If origin is discovered INSIDE the tracker, that gate must become "no explicit remote AND no discoverable origin => stealth" — and it must be ONE shared discovery, not re-probed per handler (sync/pre, prime/pre, prime/post, */post). Decide where the single discovery sits (a prime-time resolve that writes the resolved remote somewhere the binding carries? a per-op tracker helper?) before implementing. Also confirm: should mutate ops (create/claim) discover origin too, or only prime, with prime persisting the resolved remote for the session?
