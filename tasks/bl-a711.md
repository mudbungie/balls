Problem (recurring, maintainer-observed across agents; surfaced in the bl-c3c0 retro, 2026-06-28): when an agent splits one piece of work into N balls it tends to file them FLAT (no --needs/--blocks edges), silently declaring them fully independent. Under the one-agent-per-unblocked-ball dispatch that means 'claim all N in parallel, now' — colliding on shared files and ignoring an order the agent often already stated in prose (a design doc's 'foundation/first/companion', or a commit note). bl skill documents the MECHANISM (--needs ID[:OP], --blocks, --subtask-of, the blocked status, 'core enforces blockers') but never the DISCIPLINE: that filing flat IS a parallelism decision, and that a stated order must be encoded as a gate because the ball graph — not the doc — is the SSOT the dispatcher reads.

LEAD FIX: a concise imperative norm in the bl skill manual (SKILL.md at repo root — printed verbatim via include_str! in src/lib.rs:158, so just edit the file), placed near the blocker/epic discussion or the create row. Draft:
  > Splitting work? Wire the gates. Filing N balls with no edges declares them fully independent — and since one agent is dispatched per UNBLOCKED ball, that means 'all N claimed in parallel, now.' If the pieces have an order (a foundation that lands first, a companion that builds on it), encode it: --needs <id> makes a ball un-claimable until its blocker closes. 'Independent' is a deliberate choice, not the default. The order lives in the ball graph (what the dispatcher reads), never only in a design doc or commit note.
