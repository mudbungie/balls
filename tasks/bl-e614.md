## Problem

§16 migration is a 412-line throwaway Python script (`scripts/migrate-legacy.py`)
that writes legacy task state into the greenfield store **below the verb layer**.
Going below the verbs forces three smells:

- a **hand-rolled duplicate of the `task.rs` serializer** (`to_markdown`/`toml_str`
  reimplement `+++`/TOML-escape/field-order by hand) — a second representation of
  one format, drift-prone (SSOT violation);
- **orphan-ref plumbing** (`build_refs`/`subtree`/`write_tree`) to land branches
  without the real substrate;
- **id/timestamp gymnastics** — the script preserves legacy ids and historical
  `created`/`updated`, which the verbs deliberately refuse (`create` mints id =
  path; the engine stamps `now`).

It is also un-piped: balls otherwise composes beautifully with bash, but migration
can't ride `bl` verbs precisely because of the id/timestamp refusal.

## Reframe — two seams turn migration into a pure greenfield pipe

**`bl list --legacy[=REF]` / `bl show --legacy` (read side).** A bounded
compatibility flag that points the read verbs at the old `.balls/tasks/*.json`
(default `balls/tasks:.balls/tasks`) and projects it INTO the greenfield wire
shape. The legacy field-map (`claimed_by→claimant`, `depends_on→blockers`,
`type:epic→tag`, ISO→epoch, fold notes) lives read-side, behind the flag — no
external filter, no `jq`, and no legacy binary needed at cutover (the greenfield
`bl` reads its own predecessor). Severable: deleting the flag deletes the
capability with no core edit; it is removed post-cutover.

**`bl adopt` (a.k.a. `import`) (write side) — the inverse of `show --json`.**
Ingest a verbatim, fully-identified greenfield task (id + timestamps included) on
stdin and write it through the real store + serializer. `show --json` reads the
bedrock projection out; `adopt` writes the same wire shape back in. Deep, narrow,
both directions. NOT migration-specific — it is "reproduce a task that already has
an identity," the shared primitive behind federation join, restore-from-backup,
and clone-seeding. Migration becomes the once-only, empty-store application of a
capability we'd want regardless.

Pipeline (all greenfield tooling, one binary):

    bl list --legacy --json | bl adopt        # + quiesce/prime/push runbook

## What dies the moment these two seams exist

- the hand-rolled serializer — adopt writes through `task.rs` (SSOT restored);
- the ref-plumbing half — `prime` founds the store, `adopt` fills it, the tracker
  push publishes it;
- config rewriting — none: the seed already IS the migrated config (every knob
  dissolved, §16), so that step is just `prime`;
- the id/timestamp code — adopt takes them verbatim;
- the external transform + the dependency on the legacy binary — both absorbed by
  the `--legacy` read projection;
- `scripts/migrate-legacy.py` itself, and its python3 test-time dependency.

## Irreducible remainder

1. **The epic reciprocal edge** — the one cross-record reconstruction (legacy
   derived "epic waits on children" from status; greenfield needs an explicit
   claim-blocker per live child). FORK: the `--legacy` projection sees the whole
   set and can synthesize it inline, OR adopt the nodes flat then wire edges in a
   second pass with the ordinary `--needs`/`--blocks` verbs (preferred — reuses
   the real edge logic, dissolves another special case).
2. **A runbook** — quiesce → prime → pipe → force-push the colliding `balls/tasks`.
   Prose. The "refuse if a task is claimed" guard is a runbook line, not code.

## Interface to nail — decisions + one trap

- **adopt shape:** strict greenfield-task JSON on stdin, written through store +
  serializer, verbatim id/timestamps, NO mint / NO stamp / NO gate. Symmetric with
  `show --json`. (Bulk partner `bl export` if bulk-read recurs; list+show covers it.)
- **collision semantics:** adopting an id that already exists → skip / error /
  replace? Never fires for a fresh migration store; federation + restore WILL hit
  it. The real spec question — where the primitive earns its generality.
- **`--legacy` scope:** read-only projection, bounded + documented + removed
  post-cutover (footprint-demarcation, not a permanent dialect). It reads legacy
  JSON instead of greenfield TOML+++; the branch lives in the read-verb loader.
- **naming:** `adopt.rs` already means *config*-adoption on prime. Either unify
  (adopt = ingest foreign state, config or task) or call the task verb `import`.
- **trap:** adopt punches exactly the hole `create` refuses (foreign id/time).
  Justify it as a distinct primitive, not `create --id`: "reproduce an existing
  identity" ≠ "mint a new one" — the id is authoritative from the source.

## Deliverable

A design doc (living artifact) converging the open decisions, then the `--legacy`
read flag + the `adopt`/`import` verb — which retires `scripts/migrate-legacy.py`
to a runbook. Supersedes the below-the-verbs migration approach.
