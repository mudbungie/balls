## Problem

§16 migration is a 412-line throwaway Python script (`scripts/migrate-legacy.py`)
that writes legacy task state into the greenfield store **below the verb layer**.
Going below the verbs forces three smells:

- a **hand-rolled duplicate of the `task.rs` serializer** (`to_markdown`/`toml_str`
  reimplement `+++`/TOML-escape/field-order, "matching the greenfield serializer"
  by hand) — a second representation of one format, drift-prone (SSOT violation);
- **orphan-ref plumbing** (`build_refs`/`subtree`/`write_tree`, `commit-tree` +
  `GIT_INDEX_FILE` juggling) to land branches without the real substrate;
- **id/timestamp gymnastics** — the script must preserve legacy ids and historical
  `created`/`updated`, which the verbs deliberately refuse (`create` mints the id =
  path; the engine stamps `now`).

It is also un-piped: balls otherwise composes beautifully with bash, but migration
can't ride `bl` verbs precisely because of the id/timestamp refusal.

## Reframe — `bl adopt` (a.k.a. `import`) is the inverse of `bl show --json`

Add the primitive the verbs are missing: ingest a **verbatim, fully-identified
greenfield task** (id + timestamps included) on stdin and write it through the
real store + serializer. `show --json` reads the bedrock projection out; `adopt`
writes that same wire shape back in. Deep, narrow, both directions.

This is **not migration-specific** — it is "reproduce a task that already has an
identity," the shared primitive behind federation join, restore-from-backup, and
clone-seeding. Migration becomes the **once-only, empty-store application** of a
capability we'd want regardless. The special case dissolves into the general path.

Pipeline:  `OLD-bl list/show --json | transform-filter | NEW-bl adopt`
(producer is the LEGACY binary — the greenfield store is empty pre-cutover; the
transform exists because the schema difference *is* the migration).

## What dies the moment `adopt` exists

- the hand-rolled serializer — adopt writes through `task.rs` (SSOT restored);
- the entire ref-plumbing half — `prime` founds the store, `adopt` fills it, the
  tracker's normal push publishes it;
- config rewriting — there is none: the seed already IS the migrated config (every
  knob dissolved, §16), so that step is just `prime`;
- the id/timestamp code — adopt takes them verbatim.

The 412-line script collapses to a transform filter + a runbook.

## Irreducible remainder (what we still need beyond the interface)

1. **A pure field-transform filter** — rename/reshape/drop/epoch + fold notes.
   The only bespoke migration code left; git-free, serializer-free, streaming,
   unit-testable as data→data (jq-able). Keep `adopt` monolingual (greenfield
   only); all legacy knowledge stays in this disposable filter.
2. **The epic reciprocal edge** — the one cross-record reconstruction (legacy
   derived "epic waits on children" from status; greenfield needs an explicit
   claim-blocker per live child). FORK: buffer-and-enrich in the filter, OR adopt
   the nodes flat then wire edges in a second pass with the ordinary
   `--needs`/`--blocks` verbs (preferred — dissolves another special case).
3. **A source reader** — ideally legacy `bl` dumps itself; else a 5-line `git show`.
4. **A runbook** — quiesce → prime → pipe → force-push the colliding `balls/tasks`.
   Prose. The "refuse if a task is claimed" guard is a runbook line, not code.

## Interface to nail — three decisions + one trap

- **Shape:** strict greenfield-task JSON on stdin, written through store +
  serializer, verbatim id/timestamps, NO mint / NO stamp / NO gate. Symmetric with
  `show --json`. (If bulk-read recurs, `bl export` is its natural partner —
  dump/restore as a pair — but list+show covers it for now.)
- **Collision semantics:** adopting an id that already exists → skip / error /
  replace? Never fires for a fresh migration store; federation + restore WILL hit
  it. This is the real spec question — where the primitive earns its generality.
- **Naming:** `adopt.rs` already means *config*-adoption on prime. Either unify
  (adopt = ingest foreign state, config or task) or call the task verb `import`.
  Decide before the name sets.
- **Trap:** adopt punches exactly the hole `create` refuses (foreign id/time).
  Justify it as a distinct primitive, not `create --id`: "reproduce an existing
  identity" ≠ "mint a new one" — the id is authoritative from the source, not
  allocated here. That framing keeps it a capability, not a migration escape hatch.

## Deliverable

A design doc (living artifact) converging the open decisions, then the `adopt`/
`import` verb implementation — which retires `scripts/migrate-legacy.py` down to a
transform filter + runbook. Supersedes the below-the-verbs migration approach.
