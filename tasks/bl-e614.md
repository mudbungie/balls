## Problem

§16 migration is a 412-line throwaway Python script (`scripts/migrate-legacy.py`)
that writes legacy task state into the greenfield store **below the verb layer**.
That forces three smells: a hand-rolled duplicate of the `task.rs` serializer
(`to_markdown`/`toml_str`) — SSOT violation; orphan-ref plumbing
(`build_refs`/`subtree`/`write_tree`) to land branches without the real substrate;
and id/timestamp gymnastics — the script preserves legacy ids + historical
`created`/`updated`, which the verbs deliberately refuse (`create` mints id = path;
the engine stamps `now`). It is also un-piped: balls composes with bash everywhere
else, but migration can't ride the verbs precisely because of that refusal.

## Reframe — one read shim + one write primitive + one button

**(a) Legacy read shim — `bl list --legacy[=REF]` / `bl show --legacy`.** A
bounded compatibility flag that points the read verbs at the old
`.balls/tasks/*.json` (default `balls/tasks:.balls/tasks`) and projects it INTO the
greenfield wire shape. The legacy field-map (`claimed_by→claimant`,
`depends_on→blockers`, `type:epic→tag`, ISO→epoch, fold notes) lives read-side,
behind the flag — no external filter, no `jq`, no legacy binary at cutover. ONE
place owns "how to read legacy." `bl list --legacy` doubles as the **dry-run /
preview** (the old `--dry-run`, dissolved into the existing read verb). Severable:
delete the flag, no core edit; removed post-cutover.

**(b) Write primitive — `bl adopt` (a.k.a. `import`), the inverse of `show --json`.**
Ingest a verbatim, fully-identified greenfield task (id + timestamps) on stdin and
write it through the real store + serializer. `show --json` reads the bedrock
projection out; `adopt` writes the same shape back in. NOT migration-specific — it
is "reproduce a task that already has an identity," the shared primitive behind
federation join, restore-from-backup, and clone-seeding.

**(c) The button — `bl adopt --legacy[=REF]`.** Wires the whole cutover data step
automatically: read via the (a) shim → adopt every node → wire the epic reciprocal
edges. ONE command. It must be **pure orchestration over (a)+(b)** — exactly
`bl list --legacy --json | bl adopt` plus the edge pass, reachable no other way and
carrying no policy the pipe can't. That constraint is what keeps it severable and
stops it from becoming a second code path for one job (the dual-path smell).

Operator's entire data step:

    bl list --legacy            # preview what migrates (dry-run)
    bl adopt  --legacy          # commit: nodes + epic edges, in one shot
    # then the runbook: prime already seeded config; force-push balls/tasks

Composable form stays available: `bl list --legacy --json | bl adopt`.

## What dies

the hand-rolled serializer (adopt writes through `task.rs` — SSOT restored); the
ref-plumbing half (`prime` founds, `adopt` fills, the tracker pushes); config
rewriting (none — the seed IS the migrated config, §16; that step is just `prime`);
the id/timestamp code (adopt takes them verbatim); the external transform + legacy-
binary dependency (absorbed by the `--legacy` projection); the epic-edge pass as
bespoke code (the button wires it via the ordinary blocker machinery); and
`scripts/migrate-legacy.py` itself + its python3 test dep. Remainder: a **runbook**
(quiesce → prime → `adopt --legacy` → force-push the colliding `balls/tasks`; the
"refuse if a task is claimed" guard is a runbook line, not code).

## Interface to nail — decisions + one trap

- **adopt shape:** strict greenfield-task JSON on stdin, written through store +
  serializer, verbatim id/timestamps, NO mint / NO stamp / NO gate. Symmetric with
  `show --json`. (Bulk partner `bl export` if bulk-read recurs; list+show covers it.)
- **collision semantics:** adopting an id that already exists → skip / error /
  replace? Never fires for a fresh migration store; federation + restore WILL hit
  it. The real spec question — where the primitive earns its generality.
- **epic edge:** the button wires it with the normal `--needs`/`--blocks` verbs
  (each live child claim-blocks its live parent), reusing real edge logic rather
  than reimplementing it.
- **`--legacy` scope:** read-only projection, bounded + documented + removed
  post-cutover (footprint-demarcation, not a permanent dialect); the legacy-JSON
  branch lives in the read-verb loader; both list/show and the adopt button share
  this single shim.
- **naming:** `adopt.rs` already means *config*-adoption on prime. Either unify
  (adopt = ingest foreign state, config or task) or call the task verb `import`.
- **trap:** adopt punches exactly the hole `create` refuses (foreign id/time).
  Justify it as a distinct primitive, not `create --id`: "reproduce an existing
  identity" ≠ "mint a new one" — the id is authoritative from the source.

## Deliverable

A design doc (living artifact) converging the open decisions, then: the `--legacy`
read shim, the `adopt`/`import` primitive, and the `adopt --legacy` button — which
retires `scripts/migrate-legacy.py` to a runbook. Supersedes the below-the-verbs
migration approach.
