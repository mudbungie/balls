WHY: promote bl-430e from incident to principle. Every plugin effect is either BINDING — one
atomic, detectable commit point (the [bl-id] squash, the store seal, the push); retry detects
it and skips — or NON-BINDING — derived and overwritten on retry (the §14 DERIVE pattern;
merge --abort leaves nothing; capture/fold artifacts recompute). Half-merged states become
immaterial BY CONSTRUCTION, never a state to fear. Tiers 1-2 then need no rollback protocol
at all: converge-on-retry covers them. Rollback remains ONLY for effects whose binding
artifact lives in an EXTERNAL system keyed to an op that never sealed (the jira orphan
ticket: a retry mints a new id, so nothing ever converges onto the orphan — someone must
delete it). §14 inverts: converge-on-retry is the rule; rollback is the short appendix for
external effects. Code barely changes; conceptual surface shrinks.

VERIFIED knot-free (no git-surgery states), with two findings folded in:

(a) The reframe REMOVES today's scariest reset. §11's un-squash rollback (git reset --hard
HEAD~1 on the integration branch) races concurrent main movement in a shared hub — a sibling
receive between squash and rollback and the reset eats THEIR commit (§14's check-the-[bl-id]-
tag-first mitigates; dropping un-squash eliminates). A standing squash without a sealed close
is exactly the bl-430e state; retry completes it. The squash is always GATED code (the gate
runs before it), so a standing delivery is never unreviewed work.

(b) The already-delivered guard — scoped by the HANDOFF case, the one structural producer of
post-delivery commits (the solo flow never makes them: the claiming agent sees the abort and
retries, nothing stranded). bl-65e0 promises committed work/<id> WIP survives unclaim and "a
later claim + close DELIVERS it". Run that across a delivered-but-unsealed close: A's close
squashes, the post push rejects (E5), the op aborts; A unclaims or the session dies; B later
claims, the surviving work/<id> re-materializes, B commits more, B closes — and the bl-430e
skip ([bl-id]-since-fork => already delivered) silently strands B's commits, breaking the
bl-65e0 promise exactly when the prior incarnation had already squashed. (Softer variant:
the same agent misreads the post-abort error as a delivery failure, "fixes", commits,
retries.)

GUARD MECHANISM — content-containment, NOT commit-presence. Ancestry cannot distinguish
"content included in the delivery" from "added after": a forge squash-merge (bl-7bfe/bl-c6fa
— PR merged on GitHub, [bl-id] in the squash subject, the tag-scan skip is the plugin's whole
submission story) ALWAYS leaves work/<id>'s commits unmerged in the ancestry sense while
their content landed, so "tag present AND unmerged commits => abort" would false-abort every
forge close. The right predicate: [bl-id]-since-fork AND work/<id> contributes content beyond
that delivery commit (a content-merge of work/<id> into the delivery commit is NOT a no-op —
git merge-tree, compared against the delivery tree) => ABORT — "already delivered; work/<id>
carries undelivered changes — file a new task or deliver manually". Content-contained =>
skip, as today; direct and forge flows both unaffected on the happy path. One read over
facts the skip already holds; converts a silently-broken promise into a loud choice.

Interplay: claim-gated on bl-a04a (fold rigor — shared squash-path ground), wired as a real
edge, not prose.
