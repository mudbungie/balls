Add `--edit` to `bl update`: source the change from $EDITOR instead of field flags. `bl update <id> --edit` opens the task's store file tasks/<id>.md (frontmatter + body) in $EDITOR (→ $VISUAL → vi), blocking; on exit parse-validate the buffer as a Task (reuse the taskfile parse) and run it through the EXISTING update seal (stamp `updated`, §5 commit, update.post hook). Unchanged buffer = no-op (idempotent). Invalid TOML / missing required field / malformed [[blockers]] = reject with the parse error, offer re-edit or abort — NEVER commit garbage.

NOT a new verb. `bl edit` was rejected: same seal, a verb can't earn itself (maintainer, 2026-06-08). This is a flag on the update parser — no Verb enum entry, no dispatch arm, no class() wiring. `--edit` is MUTUALLY EXCLUSIVE with the field-setting flags (--title/--body/--parent/-p/-t/--no-tag/--needs/--no-needs/key=value): combining them is a usage error (they'd race over the payload) — a clean either/or, set fields OR hand-edit. Non-tty or unset $EDITOR = error "set $EDITOR or pass field flags", so agents keep using flag-driven update; --edit is the human projection only.

Motivation: whole-body --body replace forces a read-modify-write to tweak one phrase in a long body (e.g. an epic carrying migrated notes). --edit dissolves that without growing the flag-zoo. id-is-path so hand-editing can't change identity; `created` preserved, `updated` always restamped by the seal (a hand-typed value is overwritten).
