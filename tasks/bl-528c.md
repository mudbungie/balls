Two halves, one blast radius (found via an SS6 observer plugin):

1. A create.pre plugin edited a SIBLING tasks/*.md in the change worktree and the stray edit SEALED into the create commit (one commit touching two task files). SS8 step 3 says the seal "validates", but validation covers only the op-shape (exactly one new file for create) - a changed sibling with broken TOML sails through.

2. Once one task file has a parse error (duplicate key in the test), EVERY `bl list` and `bl show` fails repo-wide with the parse error - one bad ball blinds the whole store.

Fix shapes, orthogonal: (a) reads degrade per-file (skip the corrupt ball with a warning naming it) so the store stays readable; (b) seal-validate that every CHANGED tasks/*.md still parses, naming the file and the last pre plugin. (a) is the important one - corruption can also arrive by hand-edit or merge.
