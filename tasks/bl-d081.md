Ergonomics footgun: `bl conf set task-remote <url>` reads as a per-checkout setting but writes a MACHINE-WIDE value that silently shadows every other repo's own store.

**Apparent scope ≠ write scope.** You run `bl conf set task-remote` from inside a repo, and `bl conf` prints it per-checkout with source `xdg` — both read as "this checkout". But the write lands a single `remote = <url>` in `~/.config/balls/config.toml`, and the resolution ladder puts global XDG **above** each repo's own `origin`. So one satellite's setup redirects the store resolution of EVERY other repo on the machine whose store lived on its own `origin`.

**Repro (hit live 2026-06-27 wiring ~/dev/balls-adversary as a center satellite):**
1. From a satellite whose code origin isn't the center: `bl conf set task-remote git@github.com:mudbungie/balls.git`.
2. `cd` to an UNRELATED project that self-tracks — `~/dev/brazen` (origin `git@github.com:mudbungie/brazen.git`, which carries its OWN `balls/tasks`).
3. `bl conf` there now shows `task-remote git@github.com:mudbungie/balls.git  xdg` — brazen's store has been redirected to the balls center. Its own store is shadowed; the next `prime`/`sync` diverges (unrelated histories) or a `create` commingles brazen's tasks into the center. SILENT: no warning at set-time that the write is machine-wide, and none at resolution-time that a global value is overriding this repo's origin.

**Why it's a footgun, not a typo.**
- The one global value sits above per-repo `origin`, so it overrides the natural per-repo signal for ALL repos — and there is NO durable per-repo override that beats it (only the stealth sentinel `none` sits above XDG, and that means "no remote", not "use my origin").
- Cross-tracking a satellite (code in repo A, tasks in center B) has no clean per-repo mechanism today, so the global XDG looks like the obvious tool — and it's the trap. The working workaround is to repoint the satellite's `origin` at the center (code remote renamed, e.g. `gh`), which is non-obvious and itself a footgun (`git push origin` then targets the center).

**Fix direction (attack before building).**
- Clean fix already named in the design notes: a per-clone / per-binding store-remote. `binding.toml` is the documented-but-unimplemented home — `layout.rs` computes its path but nothing reads/writes it. `bl conf set task-remote` should write THERE (per-checkout), not the global XDG file; then a satellite's remote is scoped to its own checkout and can't touch an unrelated repo.
- Minimum mitigation if the per-clone field is deferred: (a) `bl conf set task-remote` announces it is writing a MACHINE-WIDE value; (b) `bl conf` provenance labels it `xdg (global)`, not bare `xdg`; (c) a resolution-time warning when a global XDG remote overrides a repo whose own `origin` differs from it ("store remote (global xdg) X overrides this repo's origin Y — intended?").

**Single source of truth:** the store remote is a per-checkout fact (which center THIS clone tracks); its authoritative home should be per-checkout, not a machine global that every checkout silently inherits. (Sibling conf-ergonomics ball: bl-78e1.)
