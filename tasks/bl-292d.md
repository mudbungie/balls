Audit 2026-06-09: every one of the 52 work/* branches in the bare repo belonged to a closed/dropped task; not one live task (except the claimed bl-430e) had one. bl close's teardown removes the worktree but leaves the work/<id> branch behind, so the branch namespace grows monotonically with every delivered task. (One-time bulk cleanup of the 52 was done by hand on 2026-06-09; this ball is the systematic fix.)

Fix: close (and drop/unclaim where they tear the worktree down) should delete the work/<id> branch as part of teardown — the squash to main has already captured the diff, so the branch carries no information after delivery (single source of truth: the delivery commit IS the record; the branch is a stale second copy).

**Why:** the worktree is the unit of work and close promises to tear it down; a branch that survives its task is exactly the kind of unowned state the architecture exists to prevent. 52 leaked branches in 4 days of dogfooding = every future repo inherits an unbounded branch leak.
