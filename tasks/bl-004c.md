Pursue the founding principle of balls: minimalism / subtraction in design and dependencies. This is DISTINCT from the dead-code audit (bl-ff89, which removes UNUSED mechanism) and the deps line-item in standards (bl-5cf3) — this one questions whether USED mechanism should exist at all.

Ask of every concept / verb / flag / config field / abstraction / module seam: does it earn its keep, or is it the general path with empty inputs (a special case that's really a missing reframe)? Can two representations of one fact collapse to a single source of truth? Is each interface deep-and-narrow — does the consumer need anything behind it? Audit the dependency tree (cargo tree / cargo-machete) for crates a few lines of std would replace, applying the footprint rule: demarcation over removal, correct-substitute test before cutting.

Deliverable: a ranked subtract-list (concept / mechanism / dep → why it can go, what replaces it) + the safe cuts applied. Coordinate with bl-ff89 and bl-5cf3 to avoid double-landing the same diff.
