PROBLEM
bl-7990 added per-command help (src/help_command.rs). Its create example is:

    bl create "review the API" --subtask-of bl-1a2b

The flag DESCRIPTION is correct post-bl-5d9a ("child of ID and gate ITS
claim"). But the EXAMPLE names a *review*, and the established convention
(SKILL.md ~line 18, architecture.md §10 'gate' bullet) is that a review gate is
--blocks close ('you can't FINISH the parent until reviewed'). Since bl-5d9a,
--subtask-of gates the parent's CLAIM, not its close. So the example uses the
claim-gating flag for the one thing the spec elsewhere says should close-gate: a
reader copying it builds a review that blocks STARTING the parent instead of
FINISHING it.

ROOT CAUSE
Collision of two near-simultaneous deliveries: bl-7990 (per-command help) and
bl-5d9a (--subtask-of gates claim) landed in the same window. The help author
folded bl-5d9a into the flag description but the example's 'review' framing now
contradicts the review-gate convention.

DECISION
Change the example to a genuine decomposed subtask whose claim-gate reads
correctly — e.g.

    bl create "wire the auth endpoint" --subtask-of bl-1a2b

Leave review/sign-off/build-gate examples to --blocks close. Confirm
help_command.rs has no other 'review'+subtask-of collision, and that the
--subtask-of flag description still reads 'gate ITS claim'.

SCOPE
- src/help_command.rs: the create example string (and any test asserting it).
- No behavior change; docs/help text only. 100% coverage gate still applies.
