Observed live TWICE during the audit-feedback sweep (2026-06-09): `bl close` squash-delivered to main, then the tracker close.post push lost a race to a concurrent agent and the op aborted — but the delivery had already landed on main. Outcomes seen: (a) bl-3bfd — retry minted an EMPTY duplicate delivery commit (25dd2e1c real + bb8d771e empty, both tagged [bl-3bfd], both published); (b) bl-07d6 — task archive un-sealed (stayed claimed) while the delivery survived and the worktree was torn down; the agent recovered by deleting work/<id> by hand before retrying (deliver no-ops on an absent branch).

ROOT-CAUSE DIAGNOSIS (from the bl-07d6 agent, verified live):
1. §14 pre-phase rollback of bl-delivery cannot resolve the id post-seal — resolve_id falls back to changed task files, but the sealed change dir is clean, so it fails "found 0" and `unsquash` SILENTLY NO-OPS on every post-abort. The delivery is never unwound, and the silent no-op hides it.
2. The retry path is not delivery-aware: it re-squashes whatever work/<id> holds against the new main — an empty diff after a landed delivery — instead of detecting the [bl-id] delivery already on main.
3. The trigger is inherent: tracker close.post push races concurrent agents; any multi-agent deployment hits it.

FIX DIRECTIONS (decide jointly; subtraction-first):
- Idempotent close (preferred frame): before delivering, query delivered_in (the derived git-log primitive already exists) for an existing [bl-id] commit on main; if present, skip delivery and resume archive/teardown/push only. Dissolves the empty-dup class and makes retry the sanctioned recovery, no new mechanism.
- Fix the silent rollback no-op regardless: resolve_id failing post-seal should at minimum emit a diagnostic naming what could not be unwound (quiet wrong-behavior is the disposition §15/bl-7110 rejected).
- Optional hardening: bounded re-fetch+re-push retry in tracker post (ETXTBSY-style) so the common race never aborts the op at all.
