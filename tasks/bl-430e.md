Observed live during the bl-3bfd delivery (2026-06-09): `bl close` squash-delivered to main, then the tracker push (close.post) was rejected by concurrent balls/tasks drift and the op aborted/rolled back — but the DELIVERY had already landed on main. The retry after `bl sync` then squashed an already-empty diff, leaving an empty duplicate delivery commit on the published main (25dd2e1c real + bb8d771e empty, both tagged [bl-3bfd]).

Two defects in the close tail ordering/atomicity:
1. The §8 anvil-atomic seal covers the STORE commit+integrate, but the close op spans store seal + main delivery + tracker push; an abort between delivery and the final seal/push is not unwound (delivery to a published main is arguably un-unwindable — but then the retry must be delivery-aware).
2. The retry path is not idempotent against a half-landed prior attempt: it happily creates an EMPTY squash commit instead of detecting the delivery already on main (delivered_in is a derived git-log query — the detection primitive already exists).

FIX DIRECTION (subtraction-first): make close idempotent rather than trying to make it more transactional — before squashing, check delivered_in for an existing [bl-id] delivery on main and skip the delivery step if present (resume semantics: complete the archive/teardown/push only). That dissolves the empty-dup class without new mechanism. Relates to bl-07d6 (claim-side seal-loser atomicity, fixed separately) and bl-ee85 (close gate).

Repro shape: two agents closing concurrently; loser of the store push race retries.
