## Summary

There is no per-command help. `bl help` lists one-line command summaries only; `bl help <cmd>` reprints that same top-level directory; `bl <cmd> --help` errors (`unexpected flag '--help'`); and an arity error like bare `bl create` ("expects exactly one positional argument") doesn't list the command's flags. So a command's own flags are undiscoverable without reading source/tests.

## Impact (real incident)

Filing bl-22dd, I needed to attach a body to a task. `create`'s visible surface looked title-only, so I went hunting for a structured-write path and landed on `import` (whose blurb, "ingest task JSON ... the inverse of `show --json`," reads like *the* rich-input tool). `import` is create-only and refused the existing id, costing several wrong turns -- when the right answer, `bl create "title" --body "..."`, existed all along and was one call away. I couldn't see `--body` because nothing surfaces it.

## Proposal (primary)

Add per-command help that lists the command's own flags + a usage line + 1-2 examples, wired to all the natural entry points:

- `bl <cmd> --help` and `bl help <cmd>` -> that command's help (not the top-level directory).
- An arity/parse error for a command should print that command's usage, not just the error.

Discoverability of `create --body`, `update --body`, `update --edit`, and the tag flag (note: `--tags` is rejected, `--tag` is accepted) all hang on this.

## Proposal (secondary -- same root: the map vs the territory)

Re-describe `import` so it stops advertising itself as the structured-write / round-trip tool. It is create-only and refuses existing ids, so `show --json | edit | import` (the implied "inverse") fails on any existing task. Suggested blurb: "bulk-create tasks from JSON (won't overwrite existing ids -- use `update` to modify)." Optionally add an explicit `import --update`/upsert if a true `show --json` -> `import` round-trip is wanted.

## Note

`import`'s error messages themselves are good (clear; the "id already held ... retire the holder or strip the record, then re-run" even gives the recovery). This is a help/affordance gap, not an error-quality one.
