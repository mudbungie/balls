Root cause: new_id(dir, known) (src/mutate_tests.rs) returns the first task_ids(dir) entry not in known. task_ids is raw UNSORTED read_dir order; create mints a RANDOM bl-<4hex> id. The test create_authors_a_ball_with_its_front_door_structure writes TWO pre-existing balls (bl-1000 + bl-9) but passes only ["bl-1000"] as known — so when readdir returns bl-9 before the new random id, new_id returns bl-9 and the assertions read the wrong ball. FS-order dependent: passes locally + on the close-time tarpaulin gate, fails on CI. Surfaced by the bl-22dd push (4639d211); the delivery-only diff does not touch this code.

Fix: new_id contract is "the id NOT among the known pre-existing set" — known must enumerate EVERY ball written before create. Exclude bl-9 too; with a complete known set the remainder is exactly one id, so readdir order is irrelevant. Document the contract on the helper.
